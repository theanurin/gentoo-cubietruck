# Gentoo On Cubietruck

## Вступ

Нещодавно друзі подарували мені декілька плат Cubietruck, і я вирішив, що це ідеальна нагода для чергового Weekend Project. Ці компактні ARM-пристрої з двоядерним процесором Allwinner A20, двома гігабайтами оперативної пам'яті та вбудованим SATA-контролером мають чудовий потенціал для домашніх серверних проектів.

Моя мета — використати ці плати як Home Network Gateway у наступному проекті. А що може бути краще для такого завдання, ніж Gentoo Linux? Ця дистрибуція дає повний контроль над системою, мінімальний оверхед та можливість налаштувати все під конкретні потреби. До того ж, компіляція під ARM — це цікавий виклик.

У цій статті я поділюся своїм досвідом встановлення Gentoo на Cubietruck: від підготовки SD-карти до першого завантаження системи. Попереджаю одразу — процес не швидкий, але результат того вартий.

### Що таке Cubietruck?

Cubietruck (також відомий як Cubieboard 3) — це одноплатний комп'ютер на базі ARM, випущений у 2013 році. Основні характеристики:

- **Процесор**: Allwinner A20 (dual-core ARM Cortex-A7, 1 GHz)
- **Оперативна пам'ять**: 2 GB DDR3
- **Накопичувач**: 8 GB NAND Flash + слот microSD + SATA II
- **Мережа**: Gigabit Ethernet + Wi-Fi 802.11 b/g/n
- **Інтерфейси**: HDMI, 2x USB 2.0, OTG, IR, UART

Чому саме Gentoo? Для мене це питання філософії та практичності. Gentoo дозволяє:

- Зібрати систему точно під потреби проекту

- Оптимізувати для конкретної архітектури ARM
- Мінімізувати споживання ресурсів
- Мати повний контроль над кожним компонентом

Готові розпочати? Тоді поїхали!

### Багатоетапне завантаження Allwinner A20

Процесор Allwinner A20 (як і більшість ARM SoC) використовує багатоетапний процес завантаження. Це необхідно через обмеження внутрішньої пам'яті та складність ініціалізації периферії.

#### Етап 0: BROM (Boot ROM)

**Що це?** Це невеликий завантажувач (близько 32 KB), зашитий у ROM процесора на етапі виробництва. Його код незмінний і виконується автоматично після подачі живлення.

**Що відбувається?**

1. **Подача живлення** → CPU починає виконання з адреси `0xFFFF0000` (адреса BROM у просторі пам'яті)
2. **Базова ініціалізація**: CPU налаштовує clock, вимикає кеші, ініціалізує внутрішню SRAM (96 KB)
3. **Пошук завантажувального пристрою** у такому порядку:
   - SD Card (MMC0)
   - NAND Flash
   - SPI NOR Flash
   - USB (FEL mode для прошивки/відновлення)
4. **Читання SPL**: BROM шукає спеціальну сигнатуру `eGON.BT0` на початку носія (зміщення 8 KB для SD карти)
5. **Завантаження у SRAM**: Знайдений SPL (до 32 KB) копіюється у внутрішню SRAM за адресою `0x00000000`
6. **Передача керування**: CPU переходить на адресу `0x00000000` і починає виконувати SPL

**Чому не можна одразу завантажити U-Boot?** Внутрішня SRAM процесора — лише 96 KB, а повний U-Boot займає 300-500 KB. Тому потрібен проміжний завантажувач.

#### Етап 1: SPL (Secondary Program Loader)

**Що це?** Це мінімалістична версія U-Boot (U-Boot SPL), яка вміщується у 32 KB і може виконуватися з SRAM без ініціалізованої DRAM.

**Що відбувається?**

1. **Ініціалізація DRAM**: Найважливіша задача SPL — налаштувати контролер DDR3 пам'яті
   - Налаштування timing параметрів
   - Калібрація шини пам'яті
   - Тестування доступності DRAM
2. **Ініціалізація MMC/SD**: Налаштування контролера SD карти для читання даних
3. **Завантаження U-Boot**: Читання повної версії U-Boot з SD карти (зазвичай з offset 40 KB) і копіювання у DRAM
4. **Передача керування**: CPU переходить на точку входу U-Boot у DRAM (наприклад, `0x4A000000`)

**Важливо**: SPL містить критичні налаштування для вашої конкретної плати (тайминги DRAM, конфігурація пінів). Саме тому SPL часто постачається виробником плати.

#### Етап 2: U-Boot (Das U-Boot)

**Що це?** Повноцінний завантажувач з підтримкою файлових систем, мережі, USB та багатьох інших функцій.

**Що відбувається?**

1. **Ініціалізація периферії**: UART (serial консоль), GPIO, Ethernet, USB, SATA, HDMI
2. **Завантаження середовища**: Читання змінних оточення (з окремого розділу або з файлу `uEnv.txt`)
3. **Автозавантаження** (якщо налаштовано):
   - Читання Device Tree Blob (DTB)
   - Завантаження ядра Linux (zImage/uImage) з файлової системи
   - Завантаження initramfs (опціонально)
   - Передача параметрів ядру через `bootargs`
4. **Передача керування ядру**: CPU переходить на точку входу ядра Linux

**Інтерактивний режим**: У U-Boot є консоль, де можна вручну виконувати команди, тестувати обладнання, завантажувати образи по мережі тощо.

### Схематично весь процес

```
Подача живлення
      ↓
┌─────────────────────────────────────┐
│ BROM (ROM у процесорі, 32 KB)       │
│ - Ініціалізація CPU, clock          │
│ - Пошук завантажувального носія     │
│ - Читання SPL (32 KB) → SRAM        │
└─────────────────────────────────────┘
      ↓
┌─────────────────────────────────────┐
│ SPL (SRAM, ~32 KB)                  │
│ - Ініціалізація DRAM                │
│ - Завантаження U-Boot → DRAM        │
└─────────────────────────────────────┘
      ↓
┌─────────────────────────────────────┐
│ U-Boot (DRAM, ~400 KB)              │
│ - Ініціалізація периферії           │
│ - Завантаження ядра/DTB → DRAM      │
│ - Передача керування ядру           │
└─────────────────────────────────────┘
      ↓
┌─────────────────────────────────────┐
│ Linux Kernel                        │
└─────────────────────────────────────┘
```

### Розкладка SD карти

Для завантаження Cubietruck SD карта повинна мати таку структуру:

```
Offset          Розмір      Вміст
------          ------      -----
0 KB            8 KB        Зарезервовано (MBR/partition table)
8 KB            32 KB       SPL (u-boot-sunxi-with-spl.bin)
40 KB           ~512 KB     U-Boot proper
544 KB          ...         Partition 1 (boot з ядром, DTB)
...             ...         Partition 2 (root filesystem)
```

**Важливо**: SPL та U-Boot записуються **перед** першим розділом, у так зване "пусте місце". Це чому перший розділ зазвичай починається з offset 1 MB або 2 MB.

### Практичний висновок

Коли ми готуватимемо SD карту, нам потрібно:

1. **Записати SPL+U-Boot** на початок SD карти (використовуючи `dd`)
2. **Створити розділи** для boot та root
3. **Налаштувати U-Boot** для завантаження нашого ядра

Тепер, розуміючи цей процес, перейдемо до практики!
